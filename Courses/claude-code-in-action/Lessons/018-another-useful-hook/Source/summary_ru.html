Кроме hooks PreToolUse и PostToolUse, которые обсуждались в этом курсе, есть и другие:

  • Notification — запускается, когда Claude Code отправляет уведомление: например, когда Claude нужно разрешение на использование инструмента или когда Claude Code бездействовал 60 секунд
  • Stop — запускается, когда Claude Code завершил ответ
  • SubagentStop — запускается, когда завершился сабагент (в интерфейсе он отображается как «Task»)
  • PreCompact — запускается перед операцией compact, ручной или автоматической
  • UserPromptSubmit — запускается, когда пользователь отправляет промпт, до того как Claude начнёт его обрабатывать
  • SessionStart — запускается при старте или возобновлении сессии
  • SessionEnd — запускается при завершении сессии

И вот в чём сложность:

  1. Вход через stdin для вашей команды меняется в зависимости от типа выполняемого hook (PreToolUse, PostToolUse, Notification и т.д.)
  2. Поле tool_input внутри этого ввода тоже меняется в зависимости от инструмента, который был вызван (в случае hooks PreToolUse и PostToolUse)

Например, вот пример stdin для hook, где hook был PostToolUse и отслеживал использование инструмента TodoWrite. Для справки: это тот инструмент, который Claude использует для ведения списка задач.

{
  "session_id": "9ecf22fa-edf8-4332-ae85-b6d5456eda64",
  "transcript_path": "<path_to_transcript>",
  "hook_event_name": "PostToolUse",
  "tool_name": "TodoWrite",
  "tool_input": {
    "todos": [{ "content": "write a readme", "status": "pending", "priority": "medium", "id": "1" }]
  },
  "tool_response": {
    "oldTodos": [],
    "newTodos": [{ "content": "write a readme", "status": "pending", "priority": "medium", "id": "1" }]
  }
}

А для сравнения вот пример ввода для hook Stop:

{
  "session_id": "af9f50b6-f042-4773-b3e2-c3a4814765ce",
  "transcript_path": "<path_to_transcript>",
  "hook_event_name": "Stop",
  "stop_hook_active": false
}

Как видите, stdin для вашей команды может сильно отличаться в зависимости от самого hook (PreToolUse, PostToolUse, Stop и т.д.) и от matcher-а (в случае PreToolUse и PostToolUse). Из-за этого писать hooks бывает сложно: вы можете просто не знать точную структуру входных данных для своей команды.

Чтобы справиться с этой проблемой, можно сделать вспомогательный hook примерно так:

"PostToolUse": [ // Или "PreToolUse" или "Stop" и т.д.
  {
    "matcher": "*",
    "hooks": [
      {
        "type": "command",
        "command": "jq . > post-log.json"
      }
    ]
  },
]

Обратите внимание на команду. Она записывает входные данные этого hook в файл post-log.json, чтобы вы могли посмотреть, что именно было передано вашей команде. Так гораздо проще понять, какие данные ваша команда должна анализировать.
