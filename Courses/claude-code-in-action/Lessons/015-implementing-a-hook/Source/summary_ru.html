Понимание данных вызова инструмента

Когда Claude пытается использовать инструмент, ваш хук получает подробную информацию об этом вызове через стандартный ввод в формате JSON. Эти данные включают:

  • идентификатор сессии и путь к транскрипту;
  • имя события хука (PreToolUse в нашем случае);
  • имя инструмента (Read, Grep и т. д.);
  • параметры входных данных инструмента, включая путь к файлу.

Ваш скрипт-хук обрабатывает эти данные и может либо разрешить операцию продолжиться, либо заблокировать её, завершившись с определённым кодом.

Реализация скрипта хука

Скрипт хука должен читать данные вызова инструмента из стандартного ввода и проверять, не пытается ли Claude получить доступ к файлу .env. Вот основная логика:

async function main() {
  const chunks = [];
  for await (const chunk of process.stdin) {
    chunks.push(chunk);
  }
  
  const toolArgs = JSON.parse(Buffer.concat(chunks).toString());
  
  // Извлекаем путь к файлу, который Claude пытается прочитать
  const readPath = 
    toolArgs.tool_input?.file_path || toolArgs.tool_input?.path || "";
  
  // Проверяем, не пытается ли Claude прочитать файл .env
  if (readPath.includes('.env')) {
    console.error("You cannot read the .env file");
    process.exit(2);
  }
}

Скрипт проверяет наличие .env в пути к файлу и блокирует операцию, если он найден. Когда вы завершаете программу с кодом 2, Claude получает сообщение об ошибке и понимает, что операция была заблокирована хуком.

Тестирование хука

После сохранения конфигурации и скрипта хука перезапустите Claude Code, чтобы изменения вступили в силу. Затем проверьте его, попросив Claude прочитать ваш файл .env.

Когда Claude попытается выполнить операцию чтения, ваш хук перехватит её и вернёт сообщение об ошибке. Claude поймёт, что операция была заблокирована, и объяснит это вам, часто указывая, что доступ был остановлен хук-ом чтения.

Та же защита работает и для операций grep - если Claude попытается искать внутри файла .env, хук тоже заблокирует это действие.

Ключевые преимущества

Такой подход даёт несколько преимуществ:

  • Проактивная защита - блокирует доступ до того, как чувствительные данные будут прочитаны
  • Прозрачная работа - Claude понимает, почему операция не удалась
  • Гибкое сопоставление - работает с несколькими инструментами (read, grep и т. д.)
  • Понятная обратная связь - даёт содержательные сообщения об ошибках

Хотя этот конкретный пример сосредоточен на файлах .env, тот же подход может защитить любые чувствительные файлы или каталоги в вашем проекте. Вы можете расширить логику, чтобы проверять несколько шаблонов файлов, или реализовать более сложные правила доступа в зависимости от ваших требований к безопасности.
